Pipeline Phase 2 hardening pass

Summary
- Preserved the existing stage roles.
- Added shared pipeline primitives for logging, runtime policy, and envelope shaping.
- Added a dead-letter path for Broadcast delivery retries so permanently failing notifications stop looping forever.
- Improved consistency at major pipeline entrypoints (`umamoe/pipeline.js`, `Workshop/pipeline.js`).
- Built a source-of-truth runtime config for broadcast retry policy.

Implemented improvements
1. Unified pipeline logging
   - `core/log.js` now supports structured context while remaining backward-compatible.
   - Added `core/pipelineLogger.js` to standardize component-scoped logs.

2. Unified envelope helpers
   - Added `core/pipelineEnvelope.js` so major pipeline boundaries can return consistent success/failure envelope shapes.

3. Source-of-truth runtime config
   - Added `core/pipelineRuntime.js`.
   - Currently centralizes `BROADCAST_MAX_DELIVERY_ATTEMPTS` and `BROADCAST_RECOVERY_BATCH_SIZE`.

4. Dead-letter design for Broadcast
   - Added dead-letter state and metadata to Archive records.
   - Added history reads + attempt summaries.
   - Announcer now checks failure count and moves repeatedly failing notifications to dead-letter instead of retrying forever.
   - Archive-Transporter now skips dead-lettered records.
   - Archive incomplete-recovery excludes dead-lettered records.

5. Pipeline entrypoint consistency
   - `umamoe/pipeline.js` and `Workshop/pipeline.js` now use shared logger/envelope helpers.
   - This keeps stage roles unchanged while making boundary behavior more consistent.

New environment knobs
- `BROADCAST_MAX_DELIVERY_ATTEMPTS` (default 5)
- `BROADCAST_RECOVERY_BATCH_SIZE` (default 100)

Operational effect
- Broadcast retries are now bounded.
- Permanently failing notifications are observable and recoverable via dead-letter state instead of endless retries.
- Pipeline boundary logs are easier to correlate.
- Main orchestration stages have more consistent success/failure envelopes.

Caveat
- This pass was implemented and verified at source level only. A live Node/Railway/Discord execution test still needs to happen after deployment.
