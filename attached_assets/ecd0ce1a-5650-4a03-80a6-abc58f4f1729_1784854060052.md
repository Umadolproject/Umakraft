Pipeline architecture input + SQLite upgrade

Summary
- Preserved the existing stage roles you defined:
  - Umamoe = acquisition + validation + trusted storage
  - Refinery = transformation + compilation + product storage
  - Workshop = fabrication + validation + staging
  - Distribution = routing + Discord delivery
  - Broadcast = notification evaluation + archival delivery state
- Replaced the most fragile part of the architecture: in-memory persistence across core stages.
- Added a shared SQLite runtime using `sql.js`, then wired stage-specific SQLite adapters for Vault, Depot, Terminal, and Archive.
- Improved the Umamoe → Refinery handoff so Refiner can automatically use the previous Vault snapshot when available for real delta gains.

How I would improve the main pipeline without removing your roles
1. Keep the stage boundaries exactly as they are, but harden the handoff contracts.
   - Miner owns external fetches only.
   - Inspector owns validity only.
   - Vault owns trusted persistence and snapshot history.
   - Refiner owns derivation and delta logic.
   - Compiler owns canonical product assembly.
   - Depot owns versioned compiled products.
   - Workshop Terminal owns immutable staged deliverables.
   - Broadcast Archive owns delivery-state persistence and retry recovery.

2. Make storage durable at every critical boundary.
   - Before: Vault, Depot, Terminal, and Archive were all in-memory.
   - Impact: Railway restart could wipe trusted source data, compiled products, staged deliverables, and notification recovery state.
   - After: these boundaries can persist to SQLite, preserving the same role architecture while making the system operationally reliable.

3. Let Vault own history so Refiner can compute real deltas.
   - Added Vault snapshot storage and previous-snapshot retrieval support.
   - Updated `umamoe/pipeline.js` to ask Vault for the previous snapshot automatically.
   - This improves fan-gain accuracy without moving business logic out of Refiner.

4. Keep Broadcast restart-safe.
   - Archive now has a SQLite path for incomplete-record recovery instead of depending on process memory.
   - Broker recovery can survive restarts much more reliably.

SQLite implementation notes
- Driver approach: `sql.js` (SQLite database file persisted to disk, no native build requirement).
- Shared DB helper: `core/sqlite.js`
- Shared backend resolution: `core/storageBackend.js`
- New adapters:
  - `umamoe/Vault/adapters/sqliteAdapter.js`
  - `Refinery/Depot/adapters/sqliteAdapter.js`
  - `Workshop/Terminal/adapters/sqliteAdapter.js`
  - `Broadcast/Archive/adapters/sqliteAdapter.js`
- Backend selection:
  - global env: `UMAKRAFT_STORAGE_BACKEND=sqlite`
  - default behavior in non-test mode now resolves toward SQLite
  - DB path env: `UMAKRAFT_SQLITE_PATH` (defaults to `/data/umakraft.sqlite`)
